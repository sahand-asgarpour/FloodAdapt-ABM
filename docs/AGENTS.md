# FloodAdapt-ABM — agent guide

FloodAdapt-ABM is a lightweight agent-based simulator that turns a precomputed
[FloodAdapt](https://pypi.org/project/flood-adapt/) impact lookup table into a
Monte-Carlo time series of building-level damages and household floodproofing
decisions under a sea-level-rise (SLR) scenario. Originally a thin research
codebase (two notebooks + two modules), it is now a packaged library
(`pip install -e .`) with a unified `SimulationEngine`, pluggable decision
rules, a pytest suite and CI — see the README and the phase sections below.

> **2026-07-09 update (HYG.2):** machine-specific paths in this guide were
> replaced with env-var/relative references, and the verification bundles are
> vendored in-repo under `verification/`. Statements marked *(historical)*
> describe the pre-refactor state and are kept for context.

## Big picture

The pipeline has two stages, one per notebook:

1. **Build the lookup table** — `1_create_lookup_table.ipynb` →
   `setup_lookup_table.py::create_lookup_table`. For a given FloodAdapt site
   database and event set it:
   - enumerates the cartesian product of `events × SLR projections × strategies`
     (`no_measures` + one `floodproof_all_<k>` per `fp_height`),
   - saves the matching `Projection` / `FloodProof` / `Strategy` / `Scenario`
     objects into the FloodAdapt database (skipping any that already exist),
   - runs every scenario through FloodAdapt (which drives SFINCS + FIAT),
   - reads back `Inundation Depth` and `Total Damage` per building footprint
     and packs them into an `xarray.Dataset` with dims
     `(object_id, slr, strategy, event)`. Event frequencies are attached as
     `ds.event.attrs["freq"]`; max potential damage (structure + content) and
     primary object types as `ds.object_id.attrs`.
   - The dataset is persisted to disk as `lookup_table_<site>_<event_set>.nc`.
2. **Simulate adaptation over time** — `2_simulate_adaptation.ipynb` →
   `abm_simulator.ABMSimulator`. Given the dataset plus a time vector and an
   SLR trajectory (typically `fa.interp_slr(slr_scenario=..., year=t)`), it
   draws `no_seq` Monte-Carlo event sequences, interpolates damages along the
   SLR axis, applies the household-level floodproofing rule, and stores
   `damage_history`, `baseline_damage_history`, and `floodproofed` arrays.

There is no shared runtime state between the two stages besides the `.nc`
lookup table — keep that interface stable.

## Input data requirements

FloodAdapt-ABM requires a pre-configured [FloodAdapt](https://pypi.org/project/flood-adapt/) 
site database. The following sections detail the required structure and formats.

### FloodAdapt database structure

The database root (`DATA_DIR`) must contain a site folder (e.g., 
`charleston_beta_release`) with the following subdirectories and files:

#### Required database components

- **Events & EventSets** (`/events/`):
  - Individual flood event definitions or an `EventSet` containing multiple 
    sub-events
  - Each event/sub-event must have a `frequency` attribute in events/year
  - The `create_combinations_matrix` function copies all sub-events from an 
    `EventSet` to top-level standalone events using `shutil.copytree`
  - Re-running on the same site is safe (`dirs_exist_ok=True`)

- **Projections** (`/projections/`):
  - Sea-level rise scenarios (e.g., "NOAA High", "NOAA Intermediate")
  - Must be compatible with `fa.interp_slr(slr_scenario=..., year=...)` for 
    time-varying SLR trajectories
  - Stage 1 creates synthetic projections `SLR_0`, `SLR_1`, ... for each 
    value in the `slr` parameter array

- **Building footprints** (`/static/exposure/`):
  - FIAT exposure table with required columns:
    - `"Object ID"`: Unique building identifier (becomes `object_id` 
      coordinate)
    - `"Max Potential Damage: structure"`: Maximum structural damage value
    - `"Max Potential Damage: content"`: Maximum content damage value
    - `"Primary Object Type"`: Building classification (residential, 
      commercial, etc.)
  - Read via `fa.get_building_footprint_impacts(scenario_name)`
  - The ABM uses `max_pot_dmg = structure + content` for threshold decisions

- **Strategies** (`/strategies/`):
  - Stage 1 creates: `"no_measures"` (empty) + `"floodproof_all_<k>"` for 
    each floodproofing height
  - Stage 2 hardcodes switching from `"no_measures"` to `"floodproof_all_0"`

- **Scenarios** (`/scenarios/`):
  - Automatically generated by stage 1 as the cartesian product of 
    projections, events, and strategies
  - Naming convention: `{projection}_{event}_{strategy}`
  - Already-run scenarios (with `finished=True` in `fa.get_scenarios()`) are 
    skipped

#### External model binaries

- **SFINCS** (`sfincs.exe`): Hydrodynamic flood model (version 2.1+ tested)
- **FIAT** (`fiat.exe`): Fast Impact Assessment Tool (version 0.2.1+ tested)

Paths set in notebook with `SFINCS_BIN_PATH` and `FIAT_BIN_PATH`. FloodAdapt 
`Settings` automatically locates them if placed in `database/system/win-64/` or 
`system/linux/` depending on platform.

### Lookup table schema

Stage 1 outputs an `xarray.Dataset` saved as `lookup_table_<site>_<event_set>.nc`. 
Schema:

```python
<xarray.Dataset>
Dimensions:
  - object_id: (n_buildings,) — Building IDs from FIAT exposure
  - slr: (n_slr,) — SLR values in `unit` (feet, meters, etc.)
  - strategy: (n_strat,) — ["no_measures", "floodproof_all_0", ...]
  - event: (n_events,) — Event names from EventSet

Coordinates:
  * object_id (object_id) — building identifiers
      .attrs['max_pot_dmg']: np.ndarray shape (n_buildings,) 
          Sum of max structure + content damage per building
      .attrs['primary_object_type']: list of strings length n_buildings
          Building type classification
  * slr (slr) — SLR projection values
  * strategy (strategy) — strategy names
  * event (event) — event identifiers
      .attrs['freq']: np.ndarray shape (n_events,)
          Event frequencies in events/year

Data variables:
  - inun_depth (object_id, slr, strategy, event) float64
      Inundation depth in feet or meters per scenario
  - total_damage (object_id, slr, strategy, event) float64
      Total economic damage per scenario
```

**Extension notes:**
- To add new variables (e.g., `casualties`, `downtime`), update:
  1. `read_impacts_dataset()` data_vars dict initialization
  2. The assignment loop that populates from `gdf_impacts`
  3. `ABMSimulator.interpolate_damage_matrix()` if the ABM consumes the 
     variable
- Do NOT reorder `object_id` independently of `max_pot_dmg` — they must stay 
  aligned

## Adaptation decision logic

### Household floodproofing rule

The decision model in `ABMSimulator._simulate_damage_history` (lines 501-509) 
implements a simple threshold rule:

```python
# Decision evaluated at the END of each timestep
for ti in range(time_steps):
    # ... event damage accumulation ...
    total_damage = sum of damages from all events in year ti
    
    # Floodproofing decision (vectorized across all households):
    not_floodproofed = ~is_floodproofed
    with_pot_dmg = self.max_pot_dmg > 0
    threshold_exceeded = np.zeros(n_households, dtype=bool)
    valid = not_floodproofed & with_pot_dmg
    threshold_exceeded[valid] = (
        total_damage[valid] / self.max_pot_dmg[valid]
    ) > self.damage_threshold
    
    is_floodproofed = is_floodproofed | threshold_exceeded
```

**Rule properties:**

- **Threshold-based**: Household floodproofs iff 
  `(damage_t / max_pot_dmg) > damage_threshold`
  - Default `damage_threshold=0.3` (30% of max potential damage)
  - Configurable via `ABMSimulator(..., damage_threshold=...)`
  
- **Reactive**: Decision is based ONLY on damages experienced in the current 
  year, not forward-looking or cost-benefit optimizing

- **Irreversible**: Once `is_floodproofed[h] = True`, it stays True for all 
  remaining timesteps
  
- **Instantaneous application**: Floodproofing status changes at the END of 
  year `t`; reduced damages apply starting year `t+1`

- **Individual/myopic**: Each household decides independently; no social 
  learning, neighbor effects, or collective action

- **Single-measure**: Currently switches to `"floodproof_all_0"` (first 
  floodproof strategy). Multi-height selection would require:
  - Passing multiple strategy names to `interpolate_damage_matrix`
  - Adding logic to choose among `"floodproof_all_0"`, `"floodproof_all_1"`, 
    etc., based on damage magnitude or wealth

- **No costs or constraints**: The decision ignores:
  - Upfront floodproofing costs
  - Household wealth/income
  - Credit/financing availability
  - Government subsidies or insurance payouts
  - Physical feasibility (e.g., structure type limits)

### Baseline comparison

`run_simulation()` runs TWO complete simulations with identical event sequences:

1. **Baseline scenario** (`_compute_baseline_no_measures`):
   - Strategy = `"no_measures"` for all households, all years
   - Floodproofing disabled (`floodproofing=False`)
   - Output: `self.baseline_damage_history[seq, household, year]`

2. **Adaptive scenario** (main `_simulate_damage_history`):
   - Strategy starts as `"no_measures"`, switches to `"floodproof_all_0"` 
     after threshold
   - Floodproofing enabled (`floodproofing=True`)
   - Outputs: 
     - `self.damage_history[seq, household, year]`
     - `self.floodproofed[seq, household, year]` (boolean)

**Metrics:**
- **Avoided damage** (per year, per sequence): 
  `baseline_damage_history - damage_history`, clipped to ≥0
- **Cumulative floodproofed**: `(cumsum(floodproofed, axis=time) > 0).sum(axis=households)`

Plots (`plot_total_damage_statistics`, `plot_event_damage_timeseries`) visualize 
both actual and avoided damages as stacked bars.

### Extending the decision model

To implement alternative rules, modify `_simulate_damage_history` (lines 455-510):

**Current decision point** (lines 501-509):
```python
if floodproofing:
    floodproofed[seq_idx, :, ti] = is_floodproofed
    # --- BEGIN DECISION LOGIC ---
    not_floodproofed = ~is_floodproofed
    with_pot_dmg = self.max_pot_dmg > 0
    threshold_exceeded = np.zeros(self.n_households, dtype=bool)
    valid = not_floodproofed & with_pot_dmg
    threshold_exceeded[valid] = (
        total_damage[valid] / self.max_pot_dmg[valid]
    ) > self.damage_threshold
    is_floodproofed = is_floodproofed | threshold_exceeded
    # --- END DECISION LOGIC ---
```

**Example extensions:**

1. **Wealth-dependent thresholds**:
   ```python
   # Add wealth array in __init__
   self.wealth = ds_impacts.object_id.attrs.get('wealth', np.ones(n_households))
   
   # In decision logic:
   adaptive_threshold = self.damage_threshold * (1 + 0.5 * (self.wealth / self.wealth.mean()))
   threshold_exceeded[valid] = (total_damage[valid] / self.max_pot_dmg[valid]) > adaptive_threshold[valid]
   ```

2. **Cost-benefit with discounting**:
   ```python
   # Estimate NPV of future avoided damages vs. upfront cost
   floodproof_cost = self.max_pot_dmg * 0.1  # example: 10% of max damage
   avoided_future = estimate_ead_reduction(...)  # user-defined
   npv = avoided_future - floodproof_cost
   threshold_exceeded[valid] = npv[valid] > 0
   ```

3. **Neighbor effects**:
   ```python
   # Requires spatial graph of neighbors
   neighbor_adoption_rate = compute_neighbor_rate(is_floodproofed, neighbor_graph)
   social_pressure = neighbor_adoption_rate > 0.3
   threshold_exceeded[valid] = (
       ((total_damage[valid] / self.max_pot_dmg[valid]) > threshold) | social_pressure[valid]
   )
   ```

4. **Government subsidies**:
   ```python
   # Add government policy array
   subsidy_available = self.government.offers_subsidy(ti, geom_id)
   reduced_threshold = self.damage_threshold * (1 - 0.5 * subsidy_available)
   ```

**Adding external parameters:**
- New behavioral inputs (wealth, social network, policy) should be:
  1. Passed to `__init__` or loaded from `ds_impacts.object_id.attrs`
  2. Stored as instance attributes aligned with `object_id` coordinate
  3. Validated for length consistency in `__init__`

**Performance note:** Keep decision logic vectorized (NumPy boolean indexing). 
Avoid per-household loops — they're 100x slower for large `n_households`.

## Key modules

- `setup_lookup_table.py` — FloodAdapt orchestration. Notable helpers:
  - `create_combinations_matrix` mutates the database on disk: it
    `shutil.copytree`s each sub-event of an `EventSet` to a top-level event
    folder so FloodAdapt can run them as standalone events. Re-running on the
    same site is fine (uses `dirs_exist_ok=True` / `AlreadyExistsError`
    guards).
  - `_cleanup_scenario_outputs` deletes everything under
    `scenarios/output/<scen>/` whose filename does **not** contain one of the
    `keep_substring` tokens (default keeps `max_water_level_map.nc`,
    `finished.txt`, `Impacts_building_footprints`). This is destructive; the
    `clean=True` default in `run_scenarios` is intentional to keep the
    database small.
  - `read_impacts_dataset` is the only place that defines the dataset schema —
    if you add variables, update both the `data_vars` dict and the assignment
    loop, and remember to mirror them in `ABMSimulator.interpolate_damage_matrix`
    if they are consumed downstream.
- `abm_simulator.py::ABMSimulator` — pure NumPy/SciPy, no FloodAdapt import.
  - `__init__` immediately calls `generate_event_sequences()`, so the RNG is
    consumed at construction time; passing the same `seed` reproduces the
    event timing.
  - `run_simulation` is the only public entry point besides plots. It first
    runs `_compute_baseline_no_measures` (strategy `no_measures` everywhere)
    and then the adaptive run. Both paths share `_simulate_damage_history`;
    `floodproofing=False` skips the floodproofing branch.
  - The adaptation rule (`_simulate_damage_history`) is hard-coded: a
    household becomes floodproofed at the end of year *t* iff
    `total_damage_t / max_pot_dmg > damage_threshold` and
    `max_pot_dmg > 0`. Once true it stays true for the rest of the sequence
    and switches to the `floodproof_all_0` lookup. Only the *first*
    floodproof strategy in the dataset is currently used — extending to
    multiple `fp_height`s means changing this function and the strategy
    selection logic.
  - `interpolate_damage_matrix` supports `linear` / `nearest` / `cubic` /
    `floor` / `ceil`; default is `linear` and `cubic` needs ≥4 SLR points in
    the lookup. Always interpolate along the SLR axis, never along event or
    object axes.
  - `slr_damage_lookup` is a deprecated alias kept for old notebooks — don't
    add new call sites.
  - Plot methods (`plot_event_damage_timeseries`,
    `plot_total_damage_statistics`) require `run_simulation` to have been
    called and raise `RuntimeError` otherwise. They assume `self.slr_values`
    aligns with `self.times`; mismatched lengths are silently truncated.

## Conventions

- **Units are display-only.** `dmg_unit` (default `"$"`) and `slr_unit`
  (default `"feet"`) are passed straight to matplotlib labels. The numeric
  values must already be consistent with whatever was fed to FloodAdapt — the
  Charleston notebooks use `UnitTypesLength.feet` end-to-end.
- **Damage dtype.** `ABMSimulator(..., damage_dtype=np.int32)` is the default;
  integer dtypes trigger `np.rint` rounding in `_simulate_damage_history`. Use
  `np.float32`/`np.float64` if sub-dollar precision matters (memory grows as
  `no_seq * n_households * time_steps`).
- **Time axis.** `self.times` is treated as evenly spaced; `dt = times[1] -
  times[0]` is used to convert per-year event frequencies into per-step
  probabilities. Non-uniform `times` will give wrong probabilities.
- **Event filtering.** Events with `freq > 1/dt` are silently dropped from
  `generate_event_sequences`. Keep sub-event frequencies in events/year.
- **Strategy naming.** The simulator hard-codes the strings `"no_measures"`
  and `"floodproof_all_0"`. `setup_lookup_table` is the single source of truth
  for those names — change both sides together.
- **Coordinate `object_id`.** Comes from FIAT's `"Object ID"` column. The
  `max_pot_dmg` attribute is a plain NumPy array aligned with that coordinate;
  do not reorder it independently of the dataset.

   `lookup_table_<site>_ABM_<event_set>.nc` next to the notebooks.

   only for `fa.interp_slr`), point `xr.open_dataset(...)` at the `.nc` file,
   pick a `slr_scenario_name` that exists in the FloodAdapt database, then run
   `ABMSimulator(...).run_simulation()` followed by the plot methods.

Validate changes with the pytest suite (`pytest tests/ -q`, includes all
bit-parity gates) and CI (`.github/workflows/ci.yml`); for stage-1 pipeline
changes additionally re-run the notebooks on a small `no_seq` (e.g. 10) and
confirm the plot outputs. *(Historical note: before Phase 2/3 there were no

## Gotchas before editing

- `create_combinations_matrix` writes into the FloodAdapt database (copies
  event folders, saves projections/strategies). Always operate on the
  `_ABM_<event_set>` copy, never the canonical site.
- `_cleanup_scenario_outputs` can silently delete useful files if you change
  the `keep_substring` defaults — extend the list rather than replacing it.
- `ABMSimulator._simulate_damage_history` allocates
  `no_seq * n_households * time_steps` arrays *twice* (baseline + adaptive).
  For large sites/sequence counts, lower `no_seq` or switch `damage_dtype` to
  a narrower integer type before scaling up.
- `interpolate_damage_matrix` extrapolates linearly outside the lookup SLR
  range (`fill_value='extrapolate'`). If your SLR scenario exceeds the
  precomputed grid, extend the grid in stage 1 instead of relying on
  extrapolation.
- `flood-adapt==1.1.3` is pinned in `environment.yml`; the FloodAdapt object
  imports in `setup_lookup_table.py` (`EventSet`, `FloodProof`,
  `SelectionType`, `UnitfulLength`, …) track that version's public API and
  may need updating if you bump it.

## DYNAMO-M Decision Bridge Coupling Integration

When coupling with the DYNAMO-M decision bridge (`DynamoDecisionBridge`), the following deviations and requirements apply:

- **SLR Trajectories**: The coupled simulation must provide continuous SLR trajectories evaluated per year. This is natively integrated using `fa.interp_slr(slr_scenario=..., year=t)`, which feeds scalar SLR values into the bridge's damage array preparation.
- **Interpolation Methods**: The bridge supports interpolation methods for evaluating the hazard matrix between the NetCDF SLR grid points. Specifically, cubic interpolation (`cubic`) is fully supported and recommended but requires the lookup table to contain at least 4 SLR points.
- **Event-Drawing Approach**: The bridge draws events using independent Bernoulli trials for each event in the catalog, where `probability = min(frequency, 1.0)`. This allows multiple independent flood events to occur in the same year.
- **Maximum Events Per Year**: Because the Bernoulli trials can hypothetically draw many events, the bridge uses a configuration parameter `max_events_per_year` (default `4`) to physically cap the number of events a household can experience in a single year. If the draw exceeds this cap, the surplus events are dropped. The cap-selection policy is being unified (see `20260707_todo_next_steps`): historically the highest-frequency events were retained, the current example scripts retain the highest-magnitude (largest expected-damage) events, and the agreed target is **random selection without replacement** from the drawn pool (which preserves the Monte-Carlo distribution).

### SEU Decision Science Validation & Test Configurations

The bridge's Subjective Expected Utility (SEU) decision module has been validated against DYNAMO-M's native `decision_module.py` and against known-good edge cases using a **reproducible Phase-1 validation battery** (see `docs/coupling_architecture.md` Section 12.1 in the DYNAMO-M repo for the full specification and design rationale). This subsection documents the test configurations so that:

1. **Developers** can verify changes to `DecisionConfig` or `DynamoDecisionBridge.evaluate_decisions` without breaking the validated science.
2. **Integration teams** understand what "correct SEU behaviour" means and can trace regressions to their source.
3. **New team members** learn what the MVP actually validates (the 6 verifications below) and what remains as future work (the V5 gap and Phase-3 implementation).

All tests are **reproducible** (seeded, deterministic); all use a **synthetic Charleston-like lookup table** (7 return-period events RP2..RP500, SLR 0..3 ft, residential+commercial mix); and all run in a single Python session so that **native DYNAMO-M** can be called directly for cross-checks. The test harness and scripts live in the session workspace and are also distributed in `seu_verification_tests/scripts/` alongside the generated reports.

#### Configuration Specifications

**Default DecisionConfig:**
```python
from floodadapt_abm import DecisionConfig, CouplingConfig

# MVP defaults (from settings.yml, verified against bridge code):
config = CouplingConfig(
    decision=DecisionConfig(
        risk_aversion=1.0,              # σ in CRRA utility U(c) = c^(1-σ)/(1-σ)
        discount_rate=0.032,            # annual time-discounting factor
        decision_horizon=15,            # years of integrated risk
        loan_duration=16,               # mortgage amortisation period (years)
        interest_rate=0.04,             # real annual interest on adaptation loan
        adaptation_cost_fraction=0.10,  # as fraction of max_pot_dmg (default)
        expenditure_cap=0.06,           # max % of annual income for adaptation costs
        amenity_weight=1.0,             # weight on non-monetary amenity (set 0 for MVP)
        error_interval=0.0,             # stochastic error term std (set 0 for deterministic)
        income_to_wealth_ratio=4.14,    # backward-mapping agent wealth from income
        max_events_per_year=4,          # hard cap on Bernoulli-drawn events/year
        risk_perc_min=0.01,             # baseline risk perception floor
        risk_perc_max=2.0,              # ceiling after fresh flood (flood_timer=0)
    )
)
```

## Phase 4a — Live DYNAMO-M parity (`DynamoLiveRule`)

Phase 4a adds a **live** decision rule that drives the *native* DYNAMO-M
`DecisionModule` instead of the pure-NumPy kernels ported into
`_core/dynamo_decision_bridge.py`. It is a pluggable `DecisionRule` (Strategy
Pattern), so it drops straight into `SimulationEngine`.

### Purpose — automated parity oracle

`DynamoLiveRule` is the executable form of the §12.1 "cross-check native" gate.
It calls upstream `calcEU_do_nothing` / `calcEU_adapt` with the *same* arrays the
bridge assembles, so a test can assert that the ported `SEURule` has **not
drifted** from upstream. This replaces the manual parity snippet above with a
one-line, CI-runnable check.

### Why it is feasible without a full Mesa model

`calcEU_adapt` (`decision_module.py:114-368`) and `calcEU_do_nothing`
(`369-471`) are near-pure array functions — they depend only on the `@njit`
static `IterateThroughFlood` and on `self.error_terms_stay`, **not** on
`self.model` / `self.agents`. `DynamoLiveRule` therefore instantiates
`DecisionModule(agents=None, model=stub_model)` where `stub_model` is a
`SimpleNamespace` exposing just `settings['decisions']['error_interval']` and
`random_module.random_state`. It sets `error_terms_stay` directly (rather than
calling `sample_error_terms`, which needs migration regions).

### Guarded / optional dependency

The native module's top-level `from gravity_models.read_gravity_model import
read_gravity_model` only resolves when `DYNAMO-M/DYNAMO-M` is on `sys.path`.
Import is therefore **lazy and defensive**:

- Path resolution order: `dynamo_path=` constructor arg → `DYNAMO_M_PATH` env
  var → conventional default `c:\repos\DYNAMO-M\DYNAMO-M`.
- `floodadapt_abm.DYNAMO_M_AVAILABLE` (bool) is probed at import time
  **without** importing DYNAMO-M or mutating `sys.path` (it only checks that
  `decision_module.py` and `gravity_models/` exist). The real import — and the
  `sys.path.append` — happens only when a `DynamoLiveRule` is constructed.
- If DYNAMO-M is absent, constructing `DynamoLiveRule` raises
  `DynamoMNotAvailable`, but importing `floodadapt_abm` and using
  `ThresholdRule` / `SEURule` keeps working.

### Bit-parity configuration

For an exact cross-check use `error_interval=0` (⇒ `error_terms_stay==1`) and
`amenity_value=0` (the default). Under this configuration `DynamoLiveRule` and
`SEURule` agree to float32 rounding and produce **identical boolean decisions**.

### Usage

```python
from floodadapt_abm import (
    SimulationEngine, CouplingConfig, DynamoLiveRule, DYNAMO_M_AVAILABLE,
)

cfg = CouplingConfig()  # error_interval=0, amenity_value=0 by default
if DYNAMO_M_AVAILABLE:
    live = DynamoLiveRule(cfg.decision)           # or dynamo_path=r"...\DYNAMO-M\DYNAMO-M"
    engine = SimulationEngine(ds=ds, config=cfg, decision_rule=live)
    res = engine.run(slr_values, no_seq=10, seed=42)
```

## Phase 4b — Mesa-native driving (time-ownership inversion)

**Engine performance contract (2026-07-09).** Two engine capabilities were added
while closing PRE.2 and are safe to rely on:
- **Per-SLR interpolation cache.** `DynamoDecisionBridge` materializes each
  strategy's damage cube once and memoises `prepare_damage_arrays` per `(SLR,
  method)`. Repeated ticks/sequences at the same SLR reuse the interpolated
  matrices (cached arrays are read-only). Deterministic → bit-identical results;
  call `clear_interp_cache()` to free memory.
- **Parallel Monte-Carlo sequences.** `SimulationEngine.run(..., n_jobs=N)` runs
  the independent sequences across a thread pool of per-worker engine clones that
  share a pre-warmed, read-only SLR cache. `n_jobs=1` (default) is the unchanged
  sequential path; `n_jobs>1` / `-1` is **bit-identical** for deterministic
  rules (`SEURule` with `error_interval == 0`, `ThresholdRule`). Use
  `DecisionRule.clone()` if you need an isolated rule instance.

**Status: delivered (scaffold), gated.** Phase 4b inverts *who owns time*. In
Phases 3/4a the year loop lives in `SimulationEngine.run()`; in Phase 4b a small
`FloodAdaptSLRModel` advances one tick at a time via `model.step()`, mirroring the
native DYNAMO-M `SLRModel.run_model()` (`while True: self.step()`). The decision
science is untouched — the same `DecisionRule.should_adapt` seam is called, only
from a different driver.

### Feasibility / scope

The real honeybees `SLRModel` cannot be instantiated in a plain FloodAdapt-ABM
env: `honeybees` and `mesa` are not installed, and `Agents.__init__` eagerly
builds the full DYNAMO-M ecosystem (Nodes, Beaches, gravity models, GLOFRIS flood
risk, geojson study areas, spin-up). Phase 4b therefore ships a **dependency-free
mirror** of the native control flow that reuses the validated FloodAdapt-ABM
kernels. Binding the real `SLRModel` is the documented **4b-full** follow-up.

**Why mirror (not bind the real model yet):** (1) the native model is not
instantiable without the full data+geodata ecosystem; (2) Phase 4a already proved
the decision kernel (`calcEU_*`) is separable from `self.model`/`self.agents`;
(3) mirroring isolates **one** variable — time ownership — so a bit-for-bit match
against `engine.run` is a clean, attributable proof of non-breakage; (4) it
matches the docx roadmap ("heavy lift — keep OUT of the MVP; schedule only after
Phase 4a proves parity"). The corresponding native processes being mirrored are
`SLRModel.run_model()` (model.py:113-127), `Agents.step()`
(agents/__init__.py:31-40) and `Nodes.step()` (agents/nodes.py:331+); the doc
Section 3.1 reproduces these excerpts alongside the mirror.

**Will 4b-full be easy?** The *contract* migration is easy/low-risk; the
*environment* migration is the real work. Already de-risked: the
`DecisionRule.should_adapt` seam is frozen and `DynamoLiveRule` already calls the
real native `calcEU`; time-ownership inversion is gate-proven; the object graph
already mirrors native names 1:1. Remaining effort: install/pin honeybees+mesa,
stand up `Data`/`FloodRisk`/gravity/spin-up, and write the one genuinely new
adapter — **lookup-table → native `CoastalNode` arrays**. Net: an infrastructure
integration exercise, not a re-architecture. Full effort/risk table is in the
Phase-4b doc Section 8.

### Object graph (mirrors DYNAMO-M)

```
FloodAdaptSLRModel            (owns time: timestep, n_timesteps, rng)
  |-- engine: SimulationEngine    (shared kernel: events, damage, rule, state)
  |-- agents: Agents
  |        +-- regions: CoastalNodePopulation   (households == view over AgentState)
  |                       +-- step(): engine.step(year) -> record history
  +-- step(): agents.step(); timestep += 1          # one tick
      run_model(): while timestep < n_timesteps: step()   # time owned HERE
```

| Native DYNAMO-M | FloodAdapt-ABM mirror (`mesa_native.py`) |
|---|---|
| `SLRModel` / `run_model()` / `step()` | `FloodAdaptSLRModel` / `run_model()` / `step()` |
| `Agents` / `Agents.regions` (Nodes) | `Agents` / `CoastalNodePopulation` |
| `CoastalNode` arrays | `AgentState` |
| `decision_module` | `DecisionRule` (`SEURule` / `DynamoLiveRule`) |

### Non-breaking (bit-parity) gate

`run_mesa_native(engine, ...)` reproduces `engine.run(...)` **bit-for-bit** for
the same seed / SLR / `no_seq`, because both call the identical per-year kernel
with the identical RNG stream (`default_rng(seed + s)` per sequence). The change
is architectural, not numerical. Verified across 5 cases (SEURule ×3, ThresholdRule
×2). Gate: **PASS**.

### Usage

```python
from floodadapt_abm import (
    SimulationEngine, CouplingConfig, run_mesa_native, FloodAdaptSLRModel,
)
import numpy as np

engine = SimulationEngine(ds=ds, config=CouplingConfig())

# (a) drop-in: time owned by model.step() ticks, not engine.run()
res = run_mesa_native(engine, np.linspace(0, 1.5, 30), no_seq=10, seed=42)

# (b) drive time by hand, one tick at a time
model = FloodAdaptSLRModel(engine, np.linspace(0, 1.5, 20), seed=42)
model.step()          # one native tick == one year -> model.timestep == 1
model.run_model()     # finish remaining years (while-loop) -> timestep == 20
```

### Will `step()` be replaced by Mesa ticks?

Yes — but only the **driver**, not the decision contract. Today `run()` owns the
`for s / for t` loops; in 4b the native `SLRModel.step()` becomes the tick driver
and pulls damages from the FloodAdapt lookup table each tick. What stays stable is
`DecisionRule.should_adapt(...)` (arrays in → bool out). `DynamoLiveRule` is the
seam: in 4a the same `calcEU` math runs inside our loop; in 4b it runs inside
Mesa's loop. It is not that `step()` is deleted — rather *who calls* the per-year
sequence changes (engine loop → `model.step()`), while `SEURule`/`DynamoLiveRule`
remain the shared decision kernels.
